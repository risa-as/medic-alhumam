# Research & Technical Decisions: نظام إدارة متجر المستلزمات الطبية

**Phase 0 output** — قرارات تقنية تحسم نقاط "NEEDS CLARIFICATION" قبل التصميم.

## القرارات

### 1. واجهة البيانات: REST API (وليس tRPC)
- **القرار**: REST عبر Next.js Route Handlers تحت `apps/web/app/api/*`، مع توثيق العقود في `contracts/`.
- **السبب**: طلب صريح من العميل؛ REST أبسط للاستهلاك من تطبيق Electron وتطبيق هاتف مستقبلي (أي عميل HTTP)، ولا يفرض اقترانًا بأنواع TypeScript على طرف العميل.
- **البدائل المرفوضة**: tRPC (اقتران قوي بالأنواع لكنه أقل ملاءمة لعملاء متعددي المنصات وغير TS).
- **تخفيف فقدان أمان الأنواع**: تعريف كل المدخلات/المخرجات بـ Zod في `packages/core` واستخدامها للتحقق على الخادم وفي `api-client`.

### 2. المصادقة: Auth.js (NextAuth v5)
- **القرار**: Auth.js مع موفّر Credentials (مستخدم/كلمة مرور) وجلسات قائمة على قاعدة البيانات عبر Prisma Adapter؛ أدوار `ADMIN` و`CASHIER`.
- **السبب**: طلب صريح؛ متكامل أصلًا مع Next.js وVercel وPrisma، ويدعم تقييد الوصول حسب الدور (FR-021).
- **الديسكتوب**: يصادق عبر REST ويخزّن رمز جلسة/مفتاح جهاز محليًا للمزامنة دون اتصال متكرر.

### 3. الاستضافة: Vercel + Neon
- **القرار**: نشر `apps/web` على Vercel؛ قاعدة PostgreSQL على Neon مع اتصال مجمّع (pooled) للبيئة Serverless.
- **السبب**: طلب صريح؛ Neon serverless Postgres يناسب دوال Vercel، وميزة الفروع (branching) مفيدة للبيئات.
- **اعتبار**: استخدام Prisma مع Neon عبر اتصال pooled (PgBouncer) للدوال، واتصال مباشر للهجرات (`DIRECT_URL`).

### 4. استراتيجية Offline-first والمزامنة (الديسكتوب)
- **القرار**: قاعدة SQLite محلية عبر Prisma في عملية Electron الرئيسية؛ كل عملية تُكتب محليًا أولًا ثم تُدرَج في قائمة انتظار مزامنة (outbox) تُرسَل إلى `/api/sync` عند توفر الاتصال.
- **حسم التعارض**: الخادم مرجع المخزون؛ الفواتير/الديون تُرسَل كأحداث غير قابلة للتعديل (append-only) لتفادي التعارض؛ تعديلات المنتجات تتبع "آخر كتابة تفوز" مع سجل تدقيق.
- **البدائل المرفوضة**: Online-only (يوقف البيع عند انقطاع الشبكة، يخالف FR-018)؛ CRDT كامل (تعقيد زائد غير مبرَّر لفرع واحد).

### 5. المبالغ المالية والعملة
- **القرار**: تخزين المبالغ كـ `Decimal` (Prisma Decimal) لا كـ float؛ عملة واحدة من الإعدادات (الدينار العراقي افتراضًا).
- **السبب**: دقة الحسابات المالية (FR-002/FR-003) وتفادي أخطاء التقريب.

### 6. الصور والفيديو (وسائط المنتجات للمتجر)
- **القرار**: تخزين الصور والفيديو على **UploadThing** (وفّر العميل `UPLOADTHING_TOKEN`) وحفظ روابطها في قاعدة البيانات؛ فيديو واحد اختياري لكل منتج مع التحقق من الصيغة (mp4/webm) والحجم الأقصى.
- **السبب**: تفادي تضخيم قاعدة البيانات؛ تكامل سهل مع Next.js وVercel عبر حزمة `uploadthing`/`@uploadthing/react`.
- **العرض**: الصور تُعرض دائمًا في المتجر؛ الفيديو يُعرض عبر أيقونة تفتح مشغّلًا (modal) عند توفّره فقط، ولا يُعرض في تطبيق سطح المكتب.

### 9. الحقول المخصّصة لكل منتج (Custom Fields)
- **القرار**: تُخزَّن تعريفات الحقول كـ JSON على `Product.customFields` (اسم، نوع select/text، قيم، إلزامي)؛ القيم المختارة تُحفظ كنسخة لحظية في `OrderItem.selectedAttributes`.
- **السبب**: مرونة عالية دون جداول إضافية؛ سمات للطلب فقط (لا مخزون لكل متغيّر حسب القرار). الـ snapshot يحمي الطلبات القديمة من تعديل التعريفات لاحقًا.
- **البدائل المرفوضة**: جداول علائقية كاملة للـ variants (تعقيد غير مبرَّر لـ v1)؛ تتبّع مخزون لكل متغيّر (مرفوض صراحةً).

### 10. صفحة الهبوط وجلسة الزبون (Landing + Session)
- **القرار**: صفحة هبوط لكل منتج `/landing/[id]`؛ الإرسال يُنشئ طلبًا عبر `POST /api/orders` بمعرّف `clientEventId` (idempotent)، ثم تحويل لصفحة `/thank-you` بعدّاد 5 ثوانٍ يحوّل للمتجر. معلومات الزبون تُحفظ في **كوكي جلسة موقّعة** عبر Next.js، وتقرؤها صفحة تأكيد الطلب لتخطّي/تعبئة الفورمة.
- **السبب**: قمع تحويل لإعلانات السوشل ميديا بأقل احتكاك؛ الكوكي الموقّعة أكثر موثوقية من sessionStorage وتعمل عبر التبويبات.
- **منع التكرار**: `clientEventId` يمنع ازدواج الطلب عند إعادة تحميل صفحة الشكر أو الإرسال المزدوج.
- **المحافظة**: قائمة منسدلة بالمحافظات العراقية الـ18 لتوحيد البيانات.

### 7. تدويل/اتجاه الواجهة
- **القرار**: عربي افتراضي مع `dir="rtl"` على مستوى الجذر؛ مكوّنات shadcn مهيّأة لـ RTL في `packages/ui`.
- **السبب**: FR-024.

### 8. الطباعة (فواتير الديسكتوب)
- **القرار**: قالب فاتورة HTML يُطبع عبر واجهة طباعة Electron؛ دعم إعادة الطباعة من سجل الفواتير.
- **السبب**: FR-005 وحالة فشل الطابعة في Edge Cases.
